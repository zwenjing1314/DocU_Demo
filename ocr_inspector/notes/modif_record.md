## 发现问题

我的这个ocr - demo我运行完了后，发现有些pdf页面会出现以下问题：
1. 当页面的图和表格多的话，它的置信度低的词语会变多。
2. 有的特殊标点符号无法识别出来。
3. 页面的页数有时候正常bboxing，有时候会出现问题。 
4. 图片上方或下方的字母有时候无法正常bboxing。
5. 对于那些简笔的图形它会识别错成圆圈。
6. 对于45度的倾斜角度的单词无法正常识别

### OCR 增强选项

当前 demo 默认开启了几个针对上述问题的增强：

- `DPI=300`：提高 PDF 渲染分辨率，改善小字、特殊标点和页码 bbox。
- `preprocess_mode=clean`：OCR 前做灰度、自动对比度和锐化，页图输出仍保留原图尺寸。
- `ocr_padding=24`：OCR 前给页面四周补白，减少页码和靠边文字被 Tesseract 漏检。
- `enable_sparse_fallback=true`：额外使用 `psm 11` 稀疏文本补扫，补图表周围的小字、页眉页脚和页码。
- `enable_rotated_text=false`：默认关闭；页面确实有 `45°` 倾斜文字时可开启，程序会做 `+45/-45` 度补扫，并把旋转结果的 bbox 映射回原图。
- `suppress_graphic_artifacts=true`：过滤孤立圆形、表格线等疑似图形误识别，过滤结果保留在每页的 `rejected_words`。

如果某些真实的单个 `O/0` 被误过滤，可以关闭 `图形误识别过滤`；如果要处理倾斜词，再在页面表单里开启 `45° 倾斜补扫`。

## 重复提交问题与修复记录

### 问题现象

如果用户在同一个文件还没有处理完成时，连续多次点击“开始 OCR”按钮，前端会重复向 `/ocr` 发起请求，后端也会重复创建任务目录并执行 OCR。结果就是本地 `uploads/` 和 `outputs/` 下会生成许多内容相同、参数相同的重复结果，浪费时间和磁盘空间。

### 修改记录（2026-04-16）

本次针对“重复点击导致重复 OCR”做了两层保护：

1. 前端防重复提交

- 在 `web/index.html` 中给“开始 OCR”按钮增加了提交锁。
- 当 OCR 请求发出后，按钮会进入禁用状态，文案变为“OCR 处理中…”。
- 如果任务还在处理中，用户再次点击提交会被前端直接拦下，不会继续发送新请求。
- 请求结束后，无论成功还是失败，按钮都会恢复可点击状态。

2. 后端请求去重与并发保护

- 在 `app.py` 中新增请求指纹机制，按“文件内容 hash + OCR 参数”生成唯一指纹。
- 如果相同文件、相同参数的 OCR 结果已经生成过，后端会直接复用已有结果，不再重新跑一次 OCR。
- 如果相同文件、相同参数的请求当前正在处理中，后端会返回 `409`，提示“相同文件和参数的 OCR 正在处理中，请稍候。”
- 对于被拦截的重复请求，会清理已经临时创建的任务目录，避免留下无用产物。
- 在 `job_manifest.json` 中额外记录了 `request_fingerprint`，方便后续排查和追踪。

3. 返回结果补充

- 接口响应新增 `cached` 字段。
- 当前端收到复用结果时，会提示“已复用相同文件和参数的已有结果。”

4. 回归测试

- 新增 `tests/test_duplicate_ocr_submission.py`。
- 已验证两种场景：
  - 同一请求重复提交时，只执行一次 OCR，后续请求直接复用结果。
  - 同一请求在处理中再次提交时，会返回 `409`，且不会额外生成重复任务目录。

## 重复提交问题二次修复记录

### 补充问题现象

第一次修复后，虽然“开始 OCR”按钮在提交时会被禁用，但锁只存在前端内存变量里，而且只锁了提交按钮，没有锁住整个 `ocrForm`。

这会带来两个风险：

- 如果页面被重新初始化，前端内存中的提交状态会丢失，按钮锁随之失效。
- 如果用户在任务执行期间修改表单里的 OCR 参数，再次提交时，前端限制不够严格，仍然可能尝试发起新的 OCR 请求。

也就是说，单纯依赖“按钮禁用”还不够，必须同时限制整个表单，并且让后端具备更强的兜底能力。

### 本次采用的修改方案

这次采用了“方案 1 + 方案 3”的组合：

1. 方案 1：前端禁用整个 `ocrForm`

- 不再只禁用“开始 OCR”按钮。
- 在 OCR 请求发出后，将整个表单中的 `input`、`select`、`button` 等可交互控件全部禁用。
- 这样用户在任务处理中无法修改文件、DPI、语言、预处理、补扫等参数。

2. 方案 3：后端只允许同一时刻存在一个 OCR 任务

- 即使前端状态丢失、页面刷新、或者用户修改了参数重新提交，后端仍然会拒绝新的 OCR 请求。
- 只要当前已有 OCR 任务在运行，新的 `/ocr` 请求就直接返回 `409`。
- 这样可以从服务端彻底兜住“刷新页面后重新提交”的问题。

### 修改记录（2026-04-16，二次修复）

1. 前端整表单锁定

- 修改文件：`web/index.html`
- 调整 `setSubmitting()` 逻辑，不再只设置提交按钮的 `disabled`。
- 改为遍历 `form.elements`，统一禁用整个 `ocrForm` 中的所有可交互控件。
- OCR 处理中时，按钮文案仍显示为“OCR 处理中…”，但同时所有表单选项也不可编辑。
- OCR 请求结束后，无论成功还是失败，表单控件都会恢复可用状态。

2. 后端全局单任务锁

- 修改文件：`app.py`
- 新增 `_active_ocr_job_id`，用于记录当前正在运行的 OCR 任务。
- 新增 `_try_acquire_global_ocr_slot()` 和 `_release_global_ocr_slot()` 两个辅助函数。
- 当 `/ocr` 收到新请求时，会先检查是否已有任务在运行：
  - 如果已有任务在运行，直接返回 `409`，提示“当前已有 OCR 任务在处理中，请等待当前任务结束后再提交新的 OCR。”
  - 如果当前没有任务运行，才允许进入 OCR 主流程。
- 这样即使用户换了参数、换了文件，甚至刷新页面重新点提交，只要前一个 OCR 还没结束，后端都会拒绝新的请求。

3. 与原有请求指纹缓存并存

- 原来的“文件内容 hash + OCR 参数”请求指纹缓存机制仍然保留。
- 也就是说：
  - OCR 运行期间：后端用“全局单任务锁”拒绝所有新任务。
  - OCR 完成之后：如果再次提交相同文件和相同参数，则直接复用已有结果。

4. 回归测试补充

- 修改文件：`tests/test_duplicate_ocr_submission.py`
- 在原有“重复请求复用缓存”的测试基础上，补充了新的场景验证：
  - 当前已有 OCR 任务运行时，即使第二次请求修改了参数，也会被后端返回 `409` 拒绝。
  - 被拒绝的请求不会额外生成新的任务目录。
  - 整个过程中实际只会执行一次 OCR。

## Layout Reader 最小版开发记录

### 需求目标

本次是在现有 OCR demo 的基础上继续开发一个最小可用的 Layout Reader，新增以下主题能力：

- 标题
- 段落
- 列表
- 页眉页脚
- 阅读顺序

最小交付目标是：

- 把一份多段落 PDF 转成带层级的 Markdown。

过关标准是：

- 标题层级和正文顺序基本正确。
- 不把页眉页脚混进正文。

### 这次实现了什么功能

1. 标题识别

- 对短文本、较大字号文本、编号标题文本做启发式判断。
- 支持类似 `1. Overview`、`1.2 Subtitle`、`第一章 / 一、 / （一）` 这类常见标题形式。
- 会把识别结果输出成 Markdown 标题层级，如 `#`、`##`、`###`。

2. 段落拼接

- 根据 `block_num`、`par_num`、行间距、缩进关系，把同一段落的多行文字合并成一段。
- 避免每一行 OCR 文本都单独断成一段。

3. 列表识别

- 支持无序列表符号，如 `-`、`*`、`•`。
- 支持常见有序列表形式，如 `1.`、`1)`、`(1)`、`A.`、`一、`、`（一）`。
- 输出为 Markdown 列表项。

4. 页眉页脚过滤

- 检测页顶和页底区域中重复出现的文本。
- 对重复页眉、重复页脚、页码文本做过滤，不把它们并入正文 Markdown。

5. 阅读顺序整理

- 先按 block 聚合，再按位置排序。
- 对单栏文档按从上到下、从左到右排序。
- 对满足条件的双栏页面做基础双栏阅读顺序判断，优先左栏再右栏。

6. 结构化 Markdown 导出

- 每页仍然会生成 `markdown/page_xxx.md`，但内容变成结构化页面 Markdown，而不是简单 OCR 文本代码块。
- 新增整份文档级 Markdown 产物：`document.md`。
- `document.md` 作为本次“最小交付”的核心结果文件。

7. 前端与接口联动

- OCR 任务完成后，页面结果区可以直接下载 `document.md`。
- `job_manifest.json` 和接口响应中也会带上 `document_markdown` 地址。

  - 双栏页面的基础阅读顺序。

### 这次产出的结果

开发完成后，当前 demo 已经具备一个最小可用的 Layout Reader 能力：

- 可以把多段落 PDF 转成带层级的 Markdown。
- 可以输出整份文档的 `document.md`。
- 可以在一定程度上处理标题、段落、列表、页眉页脚和基础阅读顺序。

### 当前限制

这次实现是“最小可交付版本”，主要依赖启发式规则，因此当前仍有边界：

- 对复杂版式、强视觉设计文档、目录页、跨栏标题、嵌套列表、表格混排等场景，结果不一定稳定。
- 双栏阅读顺序是基础判断，不是完整的版面理解引擎。
- 标题层级也是规则推断，不是语义模型判断。

不过对一般的多段落 PDF，这一版已经可以满足“先产出基本正确的层级 Markdown”的目标。

## 中文 PDF 处理与 OCR 语言选择记录

### 问题

在使用这个 demo 处理中文 PDF 时，有两个直接问题：

- 是否需要额外下载中文相关的文字包。
- 前端页面里的 `OCR 语言` 当前是自由文本输入，不够直观，用户不容易知道应该填写什么。

### 处理方案

这次采用的方案比较直接：

1. 明确中文 OCR 依赖

- 如果要识别简体中文，需要安装 Tesseract 简体中文语言包：
  - `tesseract-ocr-chi-sim`
- 如果要识别繁体中文，需要安装 Tesseract 繁体中文语言包：
  - `tesseract-ocr-chi-tra`

2. 前端把 OCR 语言改成下拉框

- 不再让用户手动输入语言代码。
- 直接提供常用可选项，降低误填概率。

3. 前后端默认值统一

- 默认 OCR 语言改成 `chi_sim+eng`。
- 这样更适合常见的中文 PDF 场景，因为中文文档里通常也会混有英文、数字、缩写和表头。

### 修改记录

1. 修改文件：`web/index.html`

- 把原来的 `OCR 语言` 文本输入框改成了下拉框。
- 新增以下选项：
  - `简体中文 + 英文` -> `chi_sim+eng`
  - `简体中文` -> `chi_sim`
  - `英文` -> `eng`
  - `繁体中文 + 英文` -> `chi_tra+eng`
  - `繁体中文` -> `chi_tra`
- 默认选中项改成 `简体中文 + 英文`。
- 同时在表单提示文案中补充说明：
  - 中文 PDF 需要先安装 Tesseract 中文语言包。
  - 默认推荐使用“简体中文 + 英文”。

2. 修改文件：`app.py`

- 将后端 `ocr_lang` 的默认值从 `eng` 改成了 `chi_sim+eng`。
- 保证前端默认选择和后端默认处理逻辑一致。

### 实现后的效果

修改完成后，当前 demo 对中文 PDF 的使用方式更明确了：

- 用户先安装中文语言包，再上传中文 PDF 即可处理。
- 页面中不再需要手动输入 OCR 语言代码，直接从下拉框选择即可。
- 默认场景下，系统会使用 `chi_sim+eng` 处理中文混排文档。

### 当前推荐用法

- 简体中文 PDF：选择 `简体中文` 或 `简体中文 + 英文`
- 中文里夹杂英文、数字、表格表头：推荐 `简体中文 + 英文`
- 繁体中文 PDF：选择 `繁体中文` 或 `繁体中文 + 英文`

### 说明

如果系统没有安装对应的 Tesseract 中文语言包，即使页面已经能选择中文 OCR 语言，实际运行时仍然会报“找不到语言数据”之类的错误。所以“前端可选”只是入口优化，真正能否识别中文，仍然取决于系统是否安装了对应语言包。
